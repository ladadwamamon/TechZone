# خطة اختبار الجودة (QA) — متجر Nexus Store

هذه خطة اختبار يدوي شاملة لفريق ضمان الجودة قبل إطلاق الموقع. الهدف: المرور على **كل صفحة** و**كل مسار عمل (workflow)** في المتجر ولوحة التحكم، وتسجيل النتيجة.

> طريقة الاستخدام: لكل حالة اختبار، نفّذ "الخطوات" وتحقّق من "النتيجة المتوقعة"، ثم ضع الحالة: ✅ نجح / ❌ فشل / ⚠️ ملاحظة. سجّل رقم الإصدار/التاريخ والمتصفح والجهاز.

## معلومات الجلسة

| الحقل | القيمة |
|------|--------|
| التاريخ | |
| اسم المختبِر | |
| البيئة (محلي / إنتاج) | |
| المتصفح + الإصدار | |
| الجهاز (سطح مكتب / جوال) | |
| رابط المتجر | |
| رابط لوحة التحكم | |

## نطاق الاختبار

- المتجر (Storefront): التصفّح، البحث، السلة، الدفع، الحساب، الصفحات الثابتة.
- لوحة التحكم (Admin CMS): تسجيل الدخول، الصلاحيات، إدارة المنتجات/الطلبات/المحتوى/الإعدادات/الأدوار/المراقبة.
- مسارات شاملة من البداية للنهاية (E2E).
- التجاوب (Responsive) والأداء وRTL (الاتجاه من اليمين لليسار).

## 3) المتجر — مسارات العمل (E2E)

| # | المسار | الخطوات | النتيجة المتوقعة | الحالة |
|---|--------|---------|------------------|--------|
| F1 | شراء كامل كزائر | تصفّح > منتج > أضف للسلة > السلة > دفع > تأكيد | يتم إنشاء الطلب وتظهر صفحة النجاح برقم طلب | |
| F2 | شراء بكوبون خصم | أضف منتج > أدخل كوبون صالح في السلة/الدفع > أكمل | يُطبّق الخصم ويظهر في الإجمالي | |
| F3 | كوبون غير صالح | أدخل كوبون خاطئ/منتهي | رسالة خطأ واضحة دون تعطّل | |
| F4 | تسليم منتج رقمي | اطلب منتجاً رقمياً وأكمل الطلب | يصل/يظهر الكود الرقمي حسب آلية التسليم | |
| F5 | السلة تبقى محفوظة | أضف للسلة ثم حدّث الصفحة | محتوى السلة لا يضيع | |
| F6 | تتبّع طلب حقيقي | استخدم رقم طلب أُنشئ فعلاً | تظهر حالة الطلب الصحيحة | |
| F7 | البحث ثم الشراء | ابحث عن منتج > افتحه > اشترِه | المسار كامل يعمل | |
| F8 | حساب العميل | سجّل دخول > اعرض طلباتك | الطلبات المرتبطة بالحساب تظهر | |

## 4) لوحة التحكم — الصفحات (Admin CMS)

> الوصول عبر `/admin/`. تذكير: عند قاعدة بيانات جديدة، أول دخول ينشئ حساب `super_admin` الأول.

| # | الصفحة | ما يجب التحقق منه | الحالة |
|---|--------|-------------------|--------|
| A1 | الإعداد الأول | عند 0 مدراء: نموذج إنشاء أول super_admin يعمل | |
| A2 | تسجيل الدخول | دخول صحيح، خطأ عند بيانات خاطئة، الجلسة تُحفظ | |
| A3 | تسجيل الخروج | إنهاء الجلسة ومنع الوصول للصفحات المحمية | |
| A4 | لوحة المعلومات | الإحصائيات (الإيرادات بالشيكل)، آخر الطلبات، الأكثر مبيعاً | |
| A5 | الطلبات | قائمة الطلبات، تفاصيل طلب، تغيير الحالة، الأسعار بالشيكل | |
| A6 | المنتجات | إضافة/تعديل/حذف، **رفع أكثر من صورتين**، **المعاينة قبل النشر**، السعر بالشيكل | |
| A7 | الأقسام | إضافة/تعديل/حذف قسم | |
| A8 | العلامات التجارية | إضافة/تعديل/حذف براند | |
| A9 | التقييمات | عرض/قبول/حذف المراجعات | |
| A10 | المدونة | إضافة/تعديل/حذف/نشر مقال | |
| A11 | الوسائط (Media) | رفع صور، اختيارها في النماذج | |
| A12 | النشرة البريدية | عرض المشتركين/التصدير | |
| A13 | الكوبونات | إنشاء كوبون (نسبة/قيمة بالشيكل)، تفعيل/إيقاف | |
| A14 | العملاء | قائمة العملاء، الإجماليات بالشيكل | |
| A15 | الحسابات (المدراء) | إضافة مدير، **إسناد دور له** | |
| A16 | الأدوار والصلاحيات | **إنشاء دور جديد واختيار صلاحياته (مصفوفة)**، تعديل/حذف دور مخصّص | |
| A17 | الإعدادات | شريط الإعلان، **إدارة شرائح السلايدر (إضافة/تعديل/ترتيب/حذف)**، التواصل، السوشيال، المزايا | |
| A18 | مراقبة الأداء | صحة السيرفر (تشغيل/ذاكرة/طلبات/أخطاء)، مؤشرات Web Vitals | |
| A19 | سجل التدقيق (Audit) | تسجيل العمليات الحسّاسة مع المستخدم والوقت | |

## 5) لوحة التحكم — الصلاحيات (RBAC)

| # | الحالة | الخطوات | النتيجة المتوقعة | الحالة |
|---|--------|---------|------------------|--------|
| R1 | دور محدود | أنشئ دوراً يملك صلاحية المنتجات فقط، وأسنده لمدير جديد | المدير يرى قسم المنتجات فقط في الشريط الجانبي | |
| R2 | منع الوصول المباشر | بحساب محدود، افتح رابط قسم ممنوع مباشرة | يُمنع الوصول (لا يظهر المحتوى الحسّاس) | |
| R3 | تعديل صلاحيات دور | عدّل صلاحيات دور موجود | تنعكس الصلاحيات فوراً على المدراء المرتبطين به | |
| R4 | حذف دور مخصّص | احذف دوراً مخصّصاً غير مستخدم | يُحذف؛ ولا يُسمح بحذف الأدوار النظامية | |
| R5 | super_admin كامل | بحساب super_admin | كل الأقسام ظاهرة ومتاحة | |
